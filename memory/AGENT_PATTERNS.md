## 关键智能体案例分析

### 1. 智能体编排者 (agents-orchestrator)

**定位**：自主流水线管理者，负责编排多个专业智能体

**核心能力**：
- 编排完整开发流水线：PM → Architect → [开发 ↔ QA 循环] → 集成
- 持续质量循环：每个任务必须通过 QA 验证
- 自动重试逻辑：失败任务带着反馈回到开发

**工作流阶段**：
```
阶段1：项目分析与规划
阶段2：技术架构
阶段3：开发-QA 持续循环（逐任务验证）
阶段4：最终集成与验证
```

**质量门禁规则**：
- 每个任务最多 3 次尝试
- QA = PASS → 进入下一任务
- QA = FAIL → 带着反馈回到开发

### 2. 小红书运营专家 (marketing-xiaohongshu-operator)

**定位**：小红书内容运营与品牌种草策略专家

**核心能力**：
- 内容策略制定：爆款公式、标题党法则、封面吸引力
- 达人合作与投放：KOL/KOC 筛选、brief 设计、ROI 追踪
- 社区运营与口碑管理

**技术交付物**：
- 爆款笔记模板（标题/封面/正文/标签策略）
- 投放排期表（预热期→集中种草期→长尾维护期）

**关键规则**：
- 不使用违禁词（平台限流）
- 达人合作走蒲公英平台报备
- 内容必须基于真实体验

### 3. AI 工程师 (engineering-ai-engineer)

**定位**：机器学习工程师与 AI 系统架构师

**核心能力**：
- 模型开发与训练：数据管线、模型选型、实验管理
- 模型部署与服务化：量化优化、Serving 架构、A/B 测试
- LLM 应用工程：Prompt Engineering、RAG 架构、Agent 系统

**RAG 服务示例代码**：
```python
class RAGService:
    def query(self, question: str, filters: dict = None) -> dict:
        # 1. 检索相关文档
        docs = self.vector_store.search(query=question, top_k=5)
        # 2. 过滤低相关度结果
        relevant = [d for d in docs if d.score >= 0.75]
        # 3. 构建 prompt
        context = "\n\n".join(d.content for d in relevant)
        # 4. 生成回答
        response = self.llm.generate(prompt=...)
        return {"answer": response.text, "sources": [...]}
```

### 4. 工作流架构师 (specialized-workflow-architect)

**定位**：工作流设计、发现与系统流程规格说明专家

**核心规则**：
- 不只为正常路径设计（覆盖7种故障模式）
- 不跳过可观测状态（客户/运维/数据库/日志）
- 不留下未定义的交接（显式 payload schema）
- 不将不相关工作流混在一起
- 不做实现决策（定义"必须发生什么"）

**7种故障模式**：
1. 正常路径
2. 输入校验失败
3. 超时故障
4. 瞬时故障（可重试）
5. 永久故障
6. 部分故障
7. 并发冲突

## 记忆管理体系

### 文件层级

| 文件 | 用途 |
|------|------|
| SOUL.md | 身份、记忆、规则（启动时加载） |
| USER.md | 用户背景和偏好 |
| MEMORY.md | 核心信息记忆索引 |
| memory/YYYY-MM-DD.md | 每日工作日志 |

### 记忆原则

1. **每次会话全新状态**：记忆依赖这些文件延续
2. **主动记录**：发现的问题、解决方案、模式都要记录
3. **索引精简**：MEMORY.md 保持精简，详情在日志中

## 关键设计模式

### 1. 三层结构
```
SOUL.md (身份层) → AGENTS.md (能力层) → IDENTITY.md (简介层)
```

### 2. 启动自动加载
```
会话开始 → 自动读取 SOUL.md → USER.md → memory/*.md → MEMORY.md
```

### 3. 交付物模板化
每个智能体都有标准化的交付物模板，确保输出一致性

### 4. 工作流步骤化
复杂的任务分解为明确的步骤，便于执行和验证

### 5. 质量门禁
关键节点设置质量验证点，不满足不推进

## 学习总结

### 215个智能体的共性

1. **身份明确**：每个智能体有清晰的定位和专长
2. **规则具体**：关键规则具体可执行，不是泛泛而谈
3. **交付标准化**：技术交付物有明确模板
4. **记忆持久化**：通过文件机制延续上下文
5. **工作流结构化**：复杂任务分解为步骤
6. **质量把关**：有验证机制和成功标准
